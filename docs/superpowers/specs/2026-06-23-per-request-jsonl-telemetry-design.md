# per-request 構造化テレメトリ（JSONL）設計書

## 問題

既存テレメトリは各段の処理時間をメモリ集計し終了時にサマリ出力するだけで、1発話ごとの粒度はメモリ常駐にとどまる。発話ごとの各段所要時間を機械可読で永続化し、後から相関 ID で全段・全マシンを join して分析する手段が無い。段は別プロセス／別マシンに分散している。

## ゴール

1発話ごとの各段所要時間を機械可読（JSONL）で即時に永続化し、trace_id で全段・全マシンを join できるようにする。既定オフのオプトインとし、ネットワーク（UNC 共有）出力先でも起動・処理を止めない。

## 非ゴール

- 1リクエストを全段まとめた単一行（wire での stage→duration 伝搬・protos 再変更が必要なため）。
- payload サイズ・キュー待ち時間など追加メトリクスの記録。
- ローテーション/圧縮/外部送信。
- 既存の終了時サマリ・人間向け行ログの変更。

## 受入基準

- 各段の完了時に、1行＝1段（ts / trace_id / stage / dur_s / pid）が JSONL へ即時追記される。
- 出力された全行が有効な JSON で、trace_id により全段・全マシンを join できる。
- 出力先が空（既定）のときは JSONL 出力が一切行われない（オプトイン）。
- 出力先が到達不能でも起動・処理が継続し、警告のうえ JSONL 出力のみが無効化される。
- 既存の `log_file` も、到達不能な共有を指しても起動をクラッシュさせず、stdout 出力で継続する。
- 既存の終了時サマリと人間向け行ログは変更されない（非破壊追加）。
