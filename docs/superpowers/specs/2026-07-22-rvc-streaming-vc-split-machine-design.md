# RVC ストリーミング VC(録音+VC マシンと再生マシンの分離)

## 問題

現状の VC は発話単位のバッチ変換のみで、`change_voice` は発話全体を reflect-pad して全区間を一括推論するステートレス変換になっている。録音は無音区切りで発話をまるごと集約してから流すため、口の動きから変換音声が出るまで発話長ぶんの遅延が乗り、連続的な低遅延 VC ができない。加えてイベント/ルーティングモデルは自己完結な離散リクエスト(Command→WorkerInput→WorkerOutput)で、ストリーム識別子・順序番号・ブロック間状態を持たない。この構造のまま「録音と VC を行うマシン」と「再生を行うマシン」を分けつつ、発話単位の字幕生成・翻訳も維持したい。

## ゴール

- 録音+VC マシンで、無音区切りを待たずに連続ブロック単位のストリーミング VC ができる(opt-in)。
- マシン間を渡るのは変換済み音声のみで、再生マシンがそれを連続再生する。
- 発話系(録音→文字起こし→翻訳→字幕)が無改変のまま並走し、挙動が維持される。
- 連続再生がブロック境界のクリックや不自然なピッチ飛び無く成立する。
- 許容遅延は対象 GPU での実測後に確定でき、そのためにマシン間トランスポートを差し替え可能にする。
- 既定では無効で、導入前と全体挙動が変わらない。

## 非ゴール

- 発話系のルーティング/ワーカー/wire 契約の変更。発話系は現状維持。
- TTS 音声のストリーミング再生(この構成の再生マシンが出す音は VC 変換音のみ)。
- 初期リリースでの双方向ストリーミング用 protobuf(vstreamer-protos)の変更。必要になったら別途。
- 複数話者・複数同時ストリーミングセッションの同時運用。
- ストリーミング VC と発話系 VC の設定共有。設定は独立とする。
- 既存の発話単位 VC(`change_voice` 経路)の挙動変更。

## 受入基準

- 録音+VC マシンで、マイク入力が無音区切りを待たずに連続ブロック単位で VC 変換される(opt-in、既定無効)。
- 変換音声のみがマシン間を渡り、再生マシンが連続的に再生する。
- 連続再生にブロック境界由来のクリック(不連続)が無い。
- ブロック境界でピッチが不自然に飛ばない。
- ストリーミング系を有効化しても、発話系(文字起こし/翻訳/字幕)の挙動は導入前と同一である。
- ストリーミング系と発話系は同一マイクから独立にキャプチャでき、発話系は無改変で動く。
- 変換が実時間に追いつかない場合でも遅延が単調増加せず、落としたことが観測可能(テレメトリ/ログ)に記録される。
- 送信パケットの欠落・並べ替えが起きても、再生側が検出して穴埋めし、それが観測可能に記録される(無音の穴を黙って作らない)。
- マシン間トランスポート方式を設定で切り替えられ、切り替え時に VC・再生・発話系の他ロジックの変更を要しない。
- 起動時に必須リソース(VC モデル/入出力デバイス/トランスポート)が無ければ明確に失敗し、同じ判定が GUI の起動前 readiness にも反映される。
- 対象 GPU でのブロック単位 RTF/遅延が計測でき、達成可能な遅延予算を判定できる。
- ストリーミング VC の設定は発話系 VC とは独立に指定できる。
- ストリーミング系が無効なとき、本変更の前後で全体挙動は同一である。
