# sender 宛先ごと並列化＋チャネル永続再利用 設計書

## 問題

sender が宛先を完全直列に送信するため、同一発話内で字幕（playback-host）への送信が音声（localhost/vc）への送信をブロックし、音声経路の開始を遅らせる。加えて送信ごとに gRPC チャネルを張り直し、生成コストと再接続失敗（冷却時 ~2.26s）を毎回払っている。転送は総遅延の約2〜3%だが、tail 遅延と音声経路の遅延に寄与する。

## ゴール

宛先間の送信を並行化して相互ブロックを解消しつつ、同一宛先の FIFO 順序を厳守する。宛先ごとに gRPC チャネルを1本張りっぱなしにして再接続コストを排除する。

## 非ゴール

- 音声トリムの一般化（既存の SUBTITLE 限定トリムはそのまま維持）。
- silence-gate 短縮・ストリーミング認識・送信のバッチ化/圧縮/チャンク分割。
- whisper / rvc / GPU 周りの変更。

## 受入基準

- 異なる宛先への送信が相互にブロックしない（遅い宛先が他宛先の送信を止めない）。
- 同一宛先への Command は投入順＝送信順（FIFO）で送られ、音声の順序逆転が起きない。
- gRPC チャネルは宛先ごとに再利用され、送信のたびに張り直されない。
- 各宛先キューは有界で、満杯時は最古の Command が破棄され最新が残る。
- 1件の送信失敗でプロセスが停止せず、以後の送信が継続する（fire-and-forget）。
- 既存の SUBTITLE 音声トリムと、空 remote のローカルディスパッチ経路が維持される。
- 既存のルーティングテストがグリーン維持である。
