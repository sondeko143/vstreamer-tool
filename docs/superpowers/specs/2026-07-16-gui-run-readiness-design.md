# GUI: すぐ簡単に動かせる情報整理 (readiness 駆動) 設計

## 問題

GUI は多 pipeline マネージャとして動くが、「開いてから pipeline が実際に動くまで」の摩擦が大きい（対象ユーザーは自分／技術者）。

1. 起動成功に必要な項目（vc の RVC モデル、voicevox の辞書・モデル・onnxruntime、GCP/ACP 資格情報など）が起動前に見えず、Start → 即死 → 下部の生ログを読む、の往復になる。しかもこの「必須か」は config の validation では出ない — ほぼ全フィールドが既定値を持ち、起動後に worker がモデル／デバイス読込で落ちるため。
2. 起動失敗・即死の理由が下部の生ログに埋もれ、原因が読み取りにくい。
3. マシン共通の重い素材パスが、pipeline 生成時のスナップショットで各 pipeline の設定ファイルにコピー固定され、pipeline 毎に再入力になる。既存 pipeline への一括更新もできず、GUI から共通値を編集する手段も無い。
4. worker が常にフラットに並び、いまの構成に関係ある必須項目・デバイス選択が任意項目に埋もれて探しにくい。

## ゴール

- pipeline を選ぶと、その構成が起動成功に必要とする項目と各々の充足状況（満たす／満たさない）が、起動前に一覧できる。
- 未充足項目の表示から、その設定箇所へ即座に到達・フォーカスできる。
- 起動失敗・即死の理由が、生ログを開かずに目立つ形で分かる。
- マシン共通の素材パスを一箇所で編集すれば既存の全 pipeline に反映でき、かつ各 pipeline の設定ファイルは単体で起動可能なまま。
- いま有効な worker の必須項目が、任意設定に埋もれず前面に出る。

## 非ゴール

- 非技術者向けのラベル平易化・ウィザード化（対象は技術者のため）。
- subtitle / translation worker のフォーム対応。
- pause / resume / reload の実行中コントロール、リモート（マルチマシン）配線の GUI 補助（既存 GUI の非ゴールを踏襲）。
- 外部前提（VOICEROID2 の常駐、CUDA/ドライバ、デバイスが実際に音を拾うか）の自動検証・自動導入。パスは「実在するか」までの確認に留める。
- config の validation 体系そのものの変更。

## 受入基準

- [ ] pipeline 選択時、有効な worker ごとに起動成功に必要な項目とその充足状況（✓／✗）が、起動前に表示される。
- [ ] 生 TOML を手編集して有効 worker や配線を変えても、必要項目の一覧・充足判定が実際の設定に追随する。
- [ ] 未充足項目の表示から、その設定箇所へワンアクションで移動してフォーカスできる。
- [ ] 必須項目が未充足のとき Start は無効化され、未充足の件数・理由が示される。かつ明示操作で未充足のまま起動できる口が残る。
- [ ] pipeline が起動直後に異常終了した場合、その理由が生ログを開かずに目立つ形で示される。
- [ ] マシン共通の素材パスを一箇所で編集でき、その値を既存の全 pipeline に反映できる。
- [ ] 反映後も、各 pipeline の設定ファイル単体で（GUI を介さず）起動できる。
- [ ] いま有効な worker の必須項目が、任意設定より前面に表示される。
