# worker 設定不備の起動時 preflight（fail-loud の全 worker 一般化）

## 問題

enable した worker の設定不備が、現状 3 通りにばらついて扱われている。

- **起動時に落ちるが理由が不明瞭**: worker のセットアップで例外が出ると TaskGroup が全 worker を巻き込んで崩壊し、生の例外 traceback が最上位に出るだけで「どの worker のどの設定が原因か」が読み取れない。
- **黙って機能しない**: ACP の認証フィールドが空・参照先デバイスが不在などでも worker は起動し、発話ごとに一時的な外部エラーに見えるログを吐き続ける。ログからは設定ミスと気づけず、パイプラインは「動いているのに何も出ない」状態になる。
- **効かない設定値への到達不能な防御**: worker_type の Enum など、実際には起きえない不備に対する防御分岐が散在する。

結果、パイプラインが起動しない／機能しない理由をログから判断できず、機能しない設定を runtime で凌ごうとする防御コードが残る。

## ゴール

- 致命的な設定不備は、worker が稼働を始める前にプロセス全体を明確な理由付きで停止する。
- 起動できたプロセスでは、その設定でその worker が機能することを前提にできる（縮退は feature 単位に限る）。
- 設定不備の失敗理由が、生の traceback ではなく、原因の worker と該当設定値を名指しする明確なログとして出る。
- 機能しない設定値を runtime で凌ぐための防御コードを減らす。

## 非ゴール

- worker 単位の隔離はしない（1 worker の致命的不備で他 worker だけ生かすことはせず、致命は必ずプロセス全体を落とす）。
- per-request の一時的な外部サービスエラー（HTTP / gRPC など）のハンドリングは変更しない。
- inbound Command の実データ検証（不正な音声データ等）は変更しない。
- 認証・モデル・デバイスの妥当性を、ネットワーク接続や実ロードなしに完全検証はしない（実リソース取得でしか判明しない失敗は、起動時の取得失敗として扱う）。
- 新しい設定項目やチューニング値の追加はしない。

## 受入基準

- enable した worker に致命的な設定不備があるとき、いずれのタスクも稼働を始める前にプロセスが停止する。
- 致命的不備が複数あるとき、最初の 1 件で止まらず、検出できた全件がまとめてログに出る。
- 致命的不備のログは、どの worker のどの設定値が原因かを名指しし、生の例外 traceback ではない。
- ACP（AmiVoice）文字起こしで appkey / engine_uri / engine_name / service_id のいずれかが空のとき、発話を待たず起動時に停止する。
- 入力／出力デバイスが解決できないとき、発話／再生を待たず起動時に停止し、対象の設定値を名指しする。
- VAD スキップゲートが有効かつモデルが欠如しているとき、起動時に停止する（既存挙動を維持）。
- feature 単位で縮退させるべき不備（例: 録音ログの保存先が書込不可）では、プロセスを止めず、警告を出して当該 feature のみ無効化し、worker 本体は動き続ける。
- 機能しない設定値を凌ぐための到達不能な防御コードが削減されている。
- 上記を検証する、実ハードウェアに依存しない自動テストがある。
